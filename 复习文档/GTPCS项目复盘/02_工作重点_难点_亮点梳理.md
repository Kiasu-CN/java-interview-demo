# GTPCS 工作重点 / 难点 / 亮点梳理

## 1. 这个项目最主要在做什么

这套项目最主要做的是：

**把分散的生产数据和业务动作串起来，形成“数据同步 - 异常治理 - 消息通知 - 报表展示 - 运营管理”的业务闭环。**

对后端开发来说，它的价值不在页面多，而在于：

- 有持续运行的定时任务
- 有规则型 SQL 和数据治理
- 有消息链路和外部推送
- 有报表口径和导出
- 有数据库触发器 / 通知驱动的自动化流程

## 2. 为什么它不是普通 CRUD 项目

如果只是普通 CRUD，通常是“表单录入 + 列表查询 + 状态更新”。

而这套项目明显超出了这个范围：

1. 有定时任务和批处理  
   不是用户点一下按钮才运行。

2. 有规则型异常检查  
   不是简单按条件查一张表，而是组合多张业务表和多类规则。

3. 有外部系统接入  
   数据湖接口、石化通推送、时序库写入都属于系统边界之外的集成。

4. 有自动通知闭环  
   业务数据一变更，就触发消息生成和推送，不靠人工操作。

5. 有业务口径整理  
   报表不是把表直接查出来，而是要先统一日期范围、统计维度和聚合逻辑。

所以这套项目更适合包装成：

**偏工程化的业务后端项目，而不是通用后台项目。**

## 3. 主线一：异常检查与质量考核

### 工作重点

- 做定时异常检查入口
- 设计并维护异常规则 SQL
- 统一异常落表
- 提供明细查询、评分统计、人工补录

### 工作难点

- 异常来源多，规则不统一
- 有些异常是“有记录但字段异常”，有些是“根本没录”
- 既要支撑自动检查，也要支撑人工录入兜底
- 统计要支持日 / 月 / 年，不只是单日查询

### 工作亮点

- `YcsjjcServiceImpl` 不是单条链路，而是多类异常统一编排
- `YcsjjcMapper.xml` 用 `MERGE` 做幂等更新，避免重复插入
- `ZlkhMapper.xml` 把异常记录进一步转成评分体系，不停留在“报错列表”

### 体现的能力

- 业务规则抽象
- 复杂 SQL 设计
- 批处理链路组织
- 幂等思维

### 为什么值得写进简历

因为它最能证明你不是只会写 Controller，而是能把 **业务规则、数据模型、任务执行、统计评分** 串起来。

### 为什么值得在面试里重点讲

因为这条线最好展开：

- 你可以讲规则怎么拆
- 可以讲为什么要统一落表
- 可以讲为什么用 `MERGE`
- 可以讲评分怎么算

### 建议定位

- `主打`

## 4. 主线二：定时任务、数据同步与批量入库

### 工作重点

- 从数据湖分页拉取生产数据
- 解析 JSON，过滤脏数据
- 分批入库 KingBase
- 对当前表 / 班次表做汇总

### 工作难点

- 外部接口返回量大，需要分页取数
- 数据字段多，批量插入参数量容易超限
- 有些数据是历史全量同步，有些是按时间窗口增量同步
- 不同设备类型的汇总逻辑并不一致

### 工作亮点

- `Cqk*ServiceImpl` 形成了相对统一的“接口拉取 + 分页 + 解析 + 入库”模式
- 多处代码显式考虑了 KingBase 参数上限，按批次写入
- `SjhzTask` + `LljServiceImpl` / `PzjzServiceImpl` 体现了面向设备的汇总处理能力
- `GuanServiceImpl` / `JckServiceImpl` 这类实现里还能看到时序库写入，不是单库开发

### 体现的能力

- 定时任务开发
- 数据同步与清洗
- 批量写入优化
- 多数据源处理

### 为什么值得写进简历

因为它能直接对应很多 JD 里的关键词：

- 数据同步
- 批处理
- 定时任务
- SQL 优化
- 多数据源

### 为什么值得在面试里重点讲

面试官很容易顺着这条线问：

- 接口怎么分页拉
- 脏数据怎么处理
- 为什么分批写
- 如何避免重复入库

### 建议定位

- `主打`

---

## 5. 主线三：消息推送与动态调度

### 工作重点

- 管理消息表与推送状态
- 从数据库读取配置动态生成定时任务
- 按排班 / 计划变更生成消息
- 对接石化通外部推送

### 工作难点

- 不同消息类型接收人逻辑不一样
- 需要避免重复推送历史消息
- 既要支持定时生成，也要支持变更后即时推送
- 要留出手动触发和自检入口，方便排障

### 工作亮点

- `PbxxTask` 自己维护 `ThreadPoolTaskScheduler`，不是单纯 `@Scheduled` 写死时间
- `executeInsertYywxjh` 会先读变更日志、按业务键去重，再查询当前有效计划
- `executePbxxShtSendByTime` 按“类型 + 时间窗口 + 推送状态”精准找待推消息
- `XxglMapper.xml` 对 `ZBXX` 与 `SCGK` 的接收人查询策略不同，说明你处理过消息模型差异

### 体现的能力

- 调度设计能力
- 消息链路设计能力
- 状态管理能力
- 外部系统集成能力

### 为什么值得写进简历

因为这是很明显的“工程化能力”证据，尤其是：

- 动态任务
- 推送状态
- 逐人推送
- 变更驱动消息

### 为什么值得在面试里重点讲

因为这类内容比“做了通知功能”更高级，容易让面试官觉得你有链路思维。

### 建议定位

- `主打`

---

## 6. 主线四：报表、晨报与导出

### 工作重点

- 按业务口径提供报表查询接口
- 聚合多组查询结果形成晨报
- 支撑前端表格导出

### 工作难点

- 口径要稳定，不然同一个报表会前后不一致
- 日期范围、维度补齐、月累计都容易出错
- 展示和导出最终都要可用，不只是后端有数据

### 工作亮点

- `YylxMapper.xml` 用 `WITH` 和分段聚合做了较完整的口径整理
- `SccbServiceImpl` 把多块数据组装成统一晨报结构
- 前端 `exportHtmlTableUtil.js` 支持合并单元格、样式和富文本导出，不是简单 CSV

### 体现的能力

- 报表查询设计
- 业务统计口径控制
- 前后端联动交付

### 为什么值得写进简历

它适合作为补充，证明你做过更贴近业务结果的工作，而不是只做底层任务。

### 为什么值得在面试里讲

如果面试官偏业务系统方向，这条线很好用；如果偏基础后端，可以放在第二层回答。

### 建议定位

- `次主打`

---

## 7. 主线五：生产运行会、值班夜查与数据库通知

### 工作重点

- 支撑值班夜查抽查记录生成与维护
- 做随机抽取单位 / 项目 / 场景的业务规则
- 变更后自动生成消息并推送给当日值班人员

### 工作难点

- 随机不是纯随机，要满足业务约束
- 需要把夜查变更和排班接收人自动关联起来
- 要做到变更即触发，而不是靠人工刷新

### 工作亮点

- `ZbycscServiceImpl` 里有冷却期和月覆盖规则，不是简单 `Random`
- `ZbycTaskMapper.xml` 自动创建触发器与通知函数
- `ZbycChangeNotifyListener` 直接监听数据库通知并做防抖
- `ZbycNotifyTask` 将变更记录转成消息并立即调用石化通推送

### 体现的能力

- 业务流程建模
- 数据库通知机制使用
- 自动化链路设计

### 为什么值得写进简历

这是项目里最有“亮点感”的一条线，尤其适合面试时拉高印象分。

### 为什么值得在面试里讲

因为 `LISTEN / NOTIFY + trigger + message push` 这个组合很容易形成记忆点。

### 建议定位

- `亮点项`

## 8. 哪些点适合主打，哪些适合作为补充

## 最适合主打

- 异常检查与质量考核闭环
- 定时任务与数据同步链路
- 消息推送与动态调度

这三条最符合校招后端岗位对“会做事”的判断标准。

## 适合做第二层补充

- 报表 / 晨报 / 导出
- 值班夜查与数据库通知

这两条更适合在简历里写 1-2 句，在面试里展开。

## 适合做加分项，不建议主打

- 应急预案文件上传下载
- 普通列表查询类模块
- 纯展示型看板样式调整

原因不是这些没价值，而是 **它们没有前面几条那么能体现“后端工程能力”**。

## 9. 面向校招简历的最终判断

如果你的目标是：

- Java 后端开发
- 后端开发工程师
- 偏工程化方向

那这个项目最应该让面试官记住你的三个标签：

1. **我做过定时任务和批处理，不只是写接口。**
2. **我做过规则 SQL 和异常治理，不只是查表。**
3. **我做过消息闭环和自动通知，不只是页面按钮。**
