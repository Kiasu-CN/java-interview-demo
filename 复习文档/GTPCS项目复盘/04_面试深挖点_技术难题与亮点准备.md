# GTPCS 面试深挖点、技术难题与亮点准备

## 1. 面试官最可能深挖的问题

## 1. 这个项目到底是干什么的

### 推荐回答思路

- 先说不是普通后台，而是生产运营平台。
- 再说核心闭环：数据同步、异常治理、消息通知、报表展示、值班协同。
- 最后落到你负责的 2 到 3 条主线。

### 可背诵版本

GTPCS 是面向油气集输业务的生产运营平台，后端既要承接生产数据，也要支撑运营管理。我这边做得比较深的是三条线：一条是定时任务和数据同步，一条是异常检查与质量考核，还有一条是消息通知链路，包括排班提醒和计划变更推送。所以它不是普通 CRUD，而是偏数据处理和业务闭环的后端项目。

## 2. 你在项目里具体负责了什么

### 推荐回答思路

- 先讲 3 条主线
- 每条只说 1 句动作
- 不要先陷进细节

### 可背诵版本

我主要负责三类工作。第一类是定时任务和数据同步，比如储气库和设备运行数据的分页拉取、解析和分批入库。第二类是异常检查和质量考核，把多类异常统一落表，再支撑明细查询和评分统计。第三类是消息通知链路，包括排班提醒、计划变更消息生成、逐人推送和推送状态更新。

## 3. 这个项目为什么不是普通 CRUD

### 推荐回答思路

- 有任务调度
- 有规则 SQL
- 有外部集成
- 有自动通知闭环

### 可背诵版本

我觉得它最不像普通 CRUD 的地方有四个。第一，它有很多持续运行的定时任务，不是用户点按钮才触发。第二，它有较多规则型 SQL 和数据治理逻辑，不只是列表查询。第三，它接了外部数据接口和石化通推送。第四，像值班夜查和计划变更这种场景，数据变化后会自动生成消息并推送，形成业务闭环。

## 2.2 定时任务与数据同步

### 面试官可能问

- 外部数据怎么拉？
- 为什么要分页？
- 为什么要分批写入？

- 入库前会做字段判空、关键字段校验，避免无效数据

### 你可以主动提的代码点

- `CqkCqjrsjServiceImpl`
- `CqkByxjlServiceImpl`

- `CqkZksbbgztServiceImpl`

### 能打动人的表达

我不是只调接口拿数据，而是把“分页拉取、JSON 解析、关键字段过滤、批量落库”这条链路走完整了。

## 2.3 消息推送与动态调度

### 面试官可能问

- 为什么不用写死的 `@Scheduled`？
- 动态任务是怎么实现的？
- 怎么避免重复推送？
- 为什么按人单独生成消息？

### 推荐回答思路

- 写死调度不够灵活，业务配置变更需要改代码
- 所以从数据库读配置，用 `ThreadPoolTaskScheduler` 动态生成任务
- 推送前根据 `xxlx + xxsj + tszt` 查待推消息
- 按人独立消息更方便状态跟踪和精确触达

### 你可以主动提的代码点

- `PbxxTask.refreshTasks`
- `PbxxTask.executeInsertYywxjh`
- `PbxxTask.executePbxxShtSendByTime`
- `XxglMapper.xml`

### 能打动人的表达

这条链路我更关注的是“任务配置可变、消息状态可追踪、推送动作可重放”，这样线上更容易维护。

## 2.5 值班夜查与数据库通知

### 面试官可能问

- 夜查为什么说是亮点？
- `LISTEN / NOTIFY` 是怎么用的？
- 为什么不用轮询？
- 随机抽查是不是纯随机？

### 推荐回答思路

- 夜查不是单表功能，有随机规则、排班接收人、消息闭环
- 变更后通过触发器 `pg_notify` 发通知
- Java 侧监听频道，收到后做防抖，再批量处理消息
- 抽查规则还考虑了三天冷却和月覆盖，不是无脑随机

### 你可以主动提的代码点

- `ZbycscServiceImpl`
- `ZbycNotifyTask`
- `ZbycChangeNotifyListener`
- `ZbycTaskMapper.xml`

### 能打动人的表达

这条线我比较想强调的一点是，我不仅做了业务页面，还把“数据变更 -> 消息生成 -> 推送触达”做成了自动闭环。

## 3. 项目中的技术难题与解决方式

## 难题 1：异常规则来源多，后续统计难统一

### 解决方式

- 统一落到异常表 `ssc_epbp_zljcycjl`
- 通过 `MERGE` 做幂等写入
- 在上层再做质量考核和分页查询

### 面试关键词

- 统一异常模型
- 幂等入库
- 统计复用

## 难题 2：外部数据量大，KingBase 批量参数受限

### 解决方式

- 分页拉取数据湖接口
- 解析后按批次写入
- 批大小按字段数控制，避免参数过多

### 面试关键词

- 分页拉取
- 分批写入
- 数据库约束驱动实现

## 难题 3：消息推送既要灵活配置，又要防重复

### 解决方式

- 调度配置存数据库
- 用 `ThreadPoolTaskScheduler` 动态刷新任务
- 推送前按类型、时间和状态筛选
- 推送成功后更新 `tszt`

### 面试关键词

- 动态调度
- 消息状态机
- 精准推送

## 难题 4：夜查变更后要尽快通知，但不能靠高频轮询

### 解决方式

- 数据库侧创建触发器和通知函数
- Java 侧 `LISTEN` 指定频道
- 收到通知后做防抖，再执行批量消息处理

### 面试关键词

- 数据库事件驱动
- LISTEN / NOTIFY
- 防抖处理

## 5. 容易被质疑的点和防守建议

## 质疑 1：这些是不是你一个人全做的

### 防守建议

不要硬说“全是我做的”，更稳妥的说法是：

- 我这段时间主要负责这几条主线的后端实现和优化
- 有些模块是我主导推进
- 有些模块我是深度参与，尤其是接口、SQL、任务和联调部分

## 质疑 2：你说做了优化，有没有量化指标

### 防守建议

如果没有实测数据，不要编。可以这样说：

- 这部分当时更多是业务问题修复和工程优化，没有做严格压测
- 但改动前后的效果可以从减少重复推送、减少批量写入失败、提升任务可维护性来体现

## 质疑 3：为什么不用 MQ / XXL-Job / Redis

### 防守建议

- 先承认这些方案可选
- 再解释当前项目约束：已有技术栈、部署复杂度、问题规模
- 最后说明现方案已经满足当前业务

示例：

当时这个场景下，项目本身主要还是基于 Spring 定时任务和数据库能力运行。像夜查通知这种场景，使用数据库触发通知就能满足实时性和复杂度要求，没有必要额外引入 MQ 增加部署和运维成本。

## 质疑 4：石化通推送失败怎么办

### 防守建议

- 说明项目里至少有 `tszt` 状态控制
- 推送失败不会误标成功
- 进一步优化可以补重试、死信或补偿任务

## 6. 适合背诵的高频回答模板

## 模板 1：项目主线

我在这个项目里主要做了三类工作。第一类是定时任务和数据同步，负责外部生产数据的分页拉取、解析和批量入库。第二类是异常检查和质量考核，基于规则 SQL 做异常治理，并支撑后续统计和补录。第三类是消息通知链路，包括排班提醒、计划变更消息和石化通推送。

## 模板 2：技术难点

我觉得这个项目的难点不在页面，而在链路。比如数据同步要考虑分页拉取和数据库参数限制，异常治理要先统一异常模型，消息推送要考虑动态调度和重复推送控制，夜查通知还用了数据库通知机制来做自动触发。

## 模板 3：最能体现能力的点

这个项目最能体现我的地方是，我不是只做单个接口，而是能把定时任务、规则 SQL、批处理、消息通知和页面联动串成完整闭环。

## 模板 4：如果面试官问你做得最深的一点

我做得最深的应该是异常检查和消息通知这两条线。异常检查这边涉及任务编排、规则 SQL 和质量考核；消息通知这边涉及动态任务、消息落库、逐人推送和外部接口调用，比较能体现后端工程能力。

## 7. 如果继续优化项目，可以怎么说

这部分很适合回答“你还有什么改进想法”。

## 可以说的优化方向

1. 给数据同步任务补充更明确的幂等键和同步日志，方便追踪单批次结果。
2. 给消息推送链路补失败重试和补偿任务，提高可靠性。
3. 对异常治理和报表查询补更细的监控指标，比如任务耗时、异常数变化、推送成功率。
4. 对规则 SQL 做进一步抽象，减少硬编码规则散落在 XML 中的问题。
5. 如果业务继续变复杂，可以把通知和变更处理抽成更独立的事件处理模块。

## 更稳妥的回答方式

如果从现在回看，我觉得后面可以继续增强三块能力：一是把同步和推送链路做得更可观测；二是给失败场景补重试和补偿；三是把一些规则配置化，而不是继续堆在 SQL 和代码里。这样系统的可维护性会更好。
