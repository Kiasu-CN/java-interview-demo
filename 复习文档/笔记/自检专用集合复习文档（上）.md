# Java 集合复习文档

## 1. 集合体系结构 + List / Set / Queue / Map 区别

### 1.1 集合体系结构（核心）

```text
Iterable
  └─ Collection
      ├─ List
      │   ├─ ArrayList
      │   ├─ LinkedList
      │   └─ Vector
      ├─ Set
      │   ├─ HashSet
      │   ├─ LinkedHashSet
      │   └─ TreeSet
      └─ Queue
          ├─ LinkedList
          ├─ ArrayDeque
          └─ PriorityQueue

Map（独立于 Collection）
  ├─ HashMap
  ├─ LinkedHashMap
  ├─ TreeMap
  └─ ConcurrentHashMap
```

### 1.2 四者区别

1. `List`
- 有序（按插入顺序）
- 可重复
- 可通过索引访问元素
- 常见：`ArrayList`、`LinkedList`

2. `Set`
- 无重复
- 一般不支持按索引访问
- 是否有序取决于实现
  - `HashSet`：无序
  - `LinkedHashSet`：保持插入顺序
  - `TreeSet`：按排序规则有序

3. `Queue`
- 队列结构，常见是 FIFO（先进先出）
- 常用于任务排队、消息处理
- `Deque` 可双端操作，可模拟栈和队列

4. `Map`
- 键值对结构（key -> value）
- key 不可重复，value 可重复
- 用于快速按 key 查找 value
- 不属于 `Collection` 体系

### 面试口述建议（30秒版）

1. 先说集合分两大块：`Collection` 和 `Map`。
2. 再说四个接口差异：`List` 有序可重复、`Set` 去重、`Queue` 队列语义、`Map` 键值对。
3. 最后补一个实战点：动态数据优先 `ArrayList`，去重优先 `Set`，键值查询优先 `Map`。

## 2. ArrayList 和 Array（数组）的区别

`ArrayList` 和 `Array` 的核心区别是“是否可动态伸缩 + 使用方式”。

1. 长度是否可变
- `Array` 是固定长度，创建后不能改变长度。
- `ArrayList` 底层基于动态数组，容量不够会扩容，使用更灵活。

2. 存储类型
- `Array` 可以直接存基本类型（如 `int[]`）和对象。
- `ArrayList` 只能存对象，基本类型需要包装类（如 `Integer`）。

3. 类型与开发体验
- `ArrayList` 支持泛型，编译期类型检查更友好。
- `Array` 不走泛型体系，主要依赖数组本身的类型约束。

4. 操作能力
- `Array` 主要是下标访问。
- `ArrayList` 有丰富 API（`add`、`remove`、`contains`、`set` 等），更适合业务开发。

5. 创建方式
- `Array` 创建时必须指定长度。
- `ArrayList` 创建时不要求指定元素个数（可指定初始容量优化性能）。

一句话收尾：
- 元素个数固定、追求简单可控时用 `Array`；
- 元素个数不确定、需要频繁增删改查时优先 `ArrayList`。

### 2.1 ArrayList 和 LinkedList 的区别（面试精简版）

答案：
1. `ArrayList` 和 `LinkedList` 都不是线程安全的。  
2. `ArrayList` 底层是动态数组，`LinkedList` 底层是双向链表。  
3. 访问性能上，`ArrayList` 随机访问更快（`get` 通常 O(1)）；`LinkedList` 随机访问慢（O(n)）。  
4. 插删性能上，`ArrayList` 中间插删要移动元素（O(n)）；`LinkedList` 头尾插删快（O(1)），但按下标插删仍要遍历（O(n)）。  
5. 内存上，`ArrayList` 的空间浪费主要是体现在数组容量预留；`LinkedList` 则是每个节点要额外存前驱/后继指针，单节点开销更大。

## 3. 集合常用方法汇总

### 3.1 Collection 通用方法

1. `add(E e)`：添加元素
2. `remove(Object o)`：删除指定元素
3. `contains(Object o)`：是否包含
4. `size()`：元素个数
5. `isEmpty()`：是否为空
6. `clear()`：清空
7. `iterator()`：获取迭代器

### 3.2 List 常用方法

1. `add(int index, E element)`：指定位置插入
2. `get(int index)`：按索引获取
3. `set(int index, E element)`：按索引修改
4. `remove(int index)`：按索引删除
5. `indexOf(Object o)`：首次出现位置
6. `subList(int from, int to)`：截取子列表（左闭右开）

### 3.3 Set 常用方法

1. `add(E e)`：添加（重复元素不会新增）
2. `contains(Object o)`：判重常用
3. `remove(Object o)`：删除
4. `size()`：去重后元素数量

### 3.4 Queue / Deque 常用方法

1. `offer(E e)`：入队（失败返回 false）
2. `poll()`：出队（空返回 null）
3. `peek()`：查看队头（空返回 null）
4. `push(E e)`：入栈（Deque）
5. `pop()`：出栈（Deque）
6. `peekFirst()/peekLast()`：查看双端元素

### 3.5 Map 常用方法

1. `put(K key, V value)`：新增/覆盖
2. `get(Object key)`：按 key 取值
3. `getOrDefault(key, defaultV)`：不存在返回默认值
4. `containsKey(Object key)`：是否包含 key
5. `remove(Object key)`：删除键值对
6. `putIfAbsent(K key, V value)`：key 不存在才放入
7. `computeIfAbsent(K key, mappingFn)`：不存在时计算并放入
8. `merge(K key, V value, remappingFn)`：合并计数高频
9. `keySet()/values()/entrySet()`：遍历三件套

## 4. ArrayList 扩容机制（面试高频）

### 答案（精简版）

1. `ArrayList` 底层是 `Object[]`，容量不够时会自动扩容。  
2. JDK8 常见扩容策略是按 `1.5` 倍增长（`newCapacity = oldCapacity + (oldCapacity >> 1)`）。  
3. 空数组首次添加元素时，会先扩到默认容量（通常是 10）。  
4. 如果本次需要的最小容量（`minCapacity`）比 1.5 倍还大，则直接扩到 `minCapacity`。  
5. 扩容本质是 `Arrays.copyOf` 复制旧数组到新数组，时间复杂度 O(n)，所以频繁扩容有性能成本。  

### 关键词

1. 底层：`Object[]`  
2. 触发条件：`size + 1 > elementData.length`  
3. 扩容倍率：`1.5` 倍  
4. 首次添加：空数组 -> 默认容量（通常 10）  
5. 本质操作：`Arrays.copyOf`（数组拷贝）  
6. 性能点：扩容有复制成本，建议可预估容量时指定初始容量  

### 常见追问（你要会答）

1. 为什么不是每次 +1？  
- 每次 +1 会导致频繁扩容与频繁拷贝，整体性能差。  

2. 为什么不是直接翻倍？  
- 翻倍能减少扩容次数，但可能浪费更多内存；1.5 倍是时间和空间的折中。  

3. 怎么优化 ArrayList 性能？  
- 元素规模可预估时，`new ArrayList<>(expectedSize)`，减少扩容次数。

## 5. fail-fast 和 fail-safe（面试高频）

### 答案

1. `fail-fast`（快速失败）  
- 遍历集合时，如果检测到集合结构被并发修改，会立刻抛出 `ConcurrentModificationException`。  
- 常见于 `ArrayList`、`HashMap` 的迭代器。  
- 典型触发：增强 for 遍历中直接调用集合的 `add/remove`。  

2. `fail-safe`（安全失败）  
- 遍历时基于“副本”或“弱一致性视图”，不会因为并发修改立即抛 `ConcurrentModificationException`。  
- 常见于 `CopyOnWriteArrayList`、`ConcurrentHashMap` 的迭代器。  
- 代价是读到的可能不是最新实时数据。  

### 关键词

1. fail-fast：检测到结构变化 -> 立即报错  
2. fail-safe：遍历副本/弱一致性 -> 不立即报错  
3. fail-fast 常见异常：`ConcurrentModificationException`  
4. fail-safe 权衡：更安全并发遍历，但可能不是最新数据  

### 对比速记

1. 一致性  
- fail-fast：强一致倾向（发现冲突就失败）  
- fail-safe：弱一致（允许并发改动，遍历视图可能滞后）  

2. 性能  
- fail-fast：开销较小，但并发修改敏感  
- fail-safe：并发友好，但复制/维护视图有额外开销  

3. 适用场景  
- fail-fast：单线程或外部已加锁场景  
- fail-safe：并发读写场景，优先保证遍历不中断

## 6. PriorityQueue（优先队列）专题 `（重点看例题）`

### 6.1 核心知识点

1. 本质  
- `PriorityQueue` 是优先队列，元素出队顺序由优先级决定，不是先进先出。  
- Java 默认是**小顶堆**（最小元素优先出队）。  

2. 底层实现  
- 基于二叉堆（数组实现的完全二叉树）。  
- 通过“上浮/下沉”维护堆序。  

3. 时间复杂度  
- `offer`（入队）：`O(log n)`  
- `poll`（弹出堆顶）：`O(log n)`  
- `peek`（查看堆顶）：`O(1)`  
- `contains/remove(Object)`：`O(n)`  

4. 使用限制  
- 非线程安全。  
- 不允许 `null`。  
- 元素要可比较（`Comparable` 或 `Comparator`），否则可能抛 `ClassCastException`。  

5. 高频场景  
- 第 K 大/第 K 小  
- Top K 高频元素  
- 合并 K 个有序链表  
- Dijkstra 最短路径  

### 6.1.1 数组如何实现完全二叉树（关键理解）

以小顶堆为例，树：

```text
        1
      /   \
     3     5
    / \   /
   7  9  8
```

对应数组：

```text
[1, 3, 5, 7, 9, 8]
```

下标关系（0 开始）：
1. 父节点下标 `i`
2. 左孩子下标：`2*i + 1`
3. 右孩子下标：`2*i + 2`
4. 子节点找父节点：`(i - 1) / 2`

结论：
1. 堆不需要链表指针，靠数组下标公式就能定位父子节点。
2. 这就是“数组实现完全二叉树”的含义。

### 6.1.2 上浮（sift up）与下沉（sift down）

1. 上浮（插入时）
- 新元素先放数组末尾。
- 若它比父节点优先级更高（小顶堆里值更小），就与父节点交换，持续向上直到满足堆序。

示例（小顶堆）：
1. 原堆：`[1, 3, 5, 7, 9, 8]`
2. 插入 `2`：`[1, 3, 5, 7, 9, 8, 2]`
3. `2 < 5`，交换：`[1, 3, 2, 7, 9, 8, 5]`
4. `2 > 1`，停止

2. 下沉（删除堆顶时）
- 弹出堆顶后，用最后一个元素补到堆顶。
- 若它比子节点优先级低（小顶堆里值更大），就和更小的那个子节点交换，持续向下直到满足堆序。

示例（小顶堆）：
1. 原堆：`[1, 3, 2, 7, 9, 8, 5]`
2. 弹出 `1`，尾元素补顶：`[5, 3, 2, 7, 9, 8]`
3. 与子节点 `3,2` 比，和 `2` 交换：`[2, 3, 5, 7, 9, 8]`
4. 继续比较后满足堆序，结束

一句话记忆：
1. 插入后修复：上浮
2. 删除堆顶后修复：下沉

### 6.2 面试答案（精简版）

`PriorityQueue` 是基于二叉堆实现的优先队列，默认小顶堆。  
插入和弹出堆顶的复杂度是 `O(log n)`，查看堆顶是 `O(1)`。  
它不保证遍历顺序有序，只有连续 `poll()` 才体现优先级顺序。  
常用于 TopK、第K大元素、最短路径等场景。

### 6.3 关键词

1. 默认小顶堆  
2. 二叉堆 + 上浮下沉  
3. offer/poll：`O(log n)`，peek：`O(1)`  
4. 非线程安全，不支持 `null`  
5. TopK / 第K大 / 最短路径

### 6.4 基础代码模板

```java
import java.util.*;

public class PriorityQueueBasicDemo {
    public static void main(String[] args) {
        // 小顶堆（默认）
        PriorityQueue<Integer> minHeap = new PriorityQueue<>();
        minHeap.offer(5);
        minHeap.offer(1);
        minHeap.offer(3);
        System.out.println(minHeap.peek()); // 1

        // 大顶堆
        PriorityQueue<Integer> maxHeap = new PriorityQueue<>(Collections.reverseOrder());
        maxHeap.offer(5);
        maxHeap.offer(1);
        maxHeap.offer(3);
        System.out.println(maxHeap.peek()); // 5

        // 自定义比较器（按第二列升序）
        PriorityQueue<int[]> pq = new PriorityQueue<>((a, b) -> a[1] - b[1]);
        pq.offer(new int[]{1, 30});
        pq.offer(new int[]{2, 10});
        System.out.println(pq.poll()[0]); // 2
    }
}
```

### 6.5 经典例题1：第 K 大元素（LC215）

思路：维护大小为 `k` 的小顶堆，堆顶就是第 K 大。

```java
import java.util.*;

public class LC215Demo {
    public static int findKthLargest(int[] nums, int k) {
        PriorityQueue<Integer> minHeap = new PriorityQueue<>();
        for (int n : nums) {
            minHeap.offer(n);
            if (minHeap.size() > k) {
                minHeap.poll();
            }
        }
        return minHeap.peek();
    }

    public static void main(String[] args) {
        int[] nums = {3, 2, 1, 5, 6, 4};
        System.out.println(findKthLargest(nums, 2)); // 5
    }
}
```

### 6.6 经典例题2：前 K 个高频元素（LC347）

思路：`Map` 计数 + 小顶堆保留前 `k` 个频率最高元素。

```java
import java.util.*;

public class LC347Demo {
    public static int[] topKFrequent(int[] nums, int k) {
        Map<Integer, Integer> freq = new HashMap<>();
        for (int n : nums) {
            freq.merge(n, 1, Integer::sum);
        }

        PriorityQueue<Integer> minHeap = new PriorityQueue<>(Comparator.comparingInt(freq::get));
        for (int n : freq.keySet()) {
            minHeap.offer(n);
            if (minHeap.size() > k) {
                minHeap.poll();
            }
        }

        int[] res = new int[k];
        for (int i = k - 1; i >= 0; i--) {
            res[i] = minHeap.poll();
        }
        return res;
    }

    public static void main(String[] args) {
        int[] nums = {1, 1, 1, 2, 2, 3};
        System.out.println(Arrays.toString(topKFrequent(nums, 2))); // [1, 2]
    }
}
```

### 6.7 经典例题3：数据流中的第 K 大元素（LC703 思路）

思路：维护固定容量 `k` 的小顶堆，每次 `add` 后堆顶即答案。

```java
import java.util.*;

class KthLargest {
    private final int k;
    private final PriorityQueue<Integer> minHeap = new PriorityQueue<>();

    public KthLargest(int k, int[] nums) {
        this.k = k;
        for (int n : nums) {
            add(n);
        }
    }

    public int add(int val) {
        minHeap.offer(val);
        if (minHeap.size() > k) {
            minHeap.poll();
        }
        return minHeap.peek();
    }
}
```

### 6.8 常见坑

1. `PriorityQueue` 直接遍历不是有序输出。  
2. 只有反复 `poll()` 才能按优先级顺序取元素。  
3. 比较器写反会导致小顶堆/大顶堆逻辑颠倒。  
4. 可变对象入堆后若关键字段变化，结果可能不正确。

## 7. ArrayBlockingQueue 和 LinkedBlockingQueue 的区别

### 答案

1. 两者都实现了 `BlockingQueue`，都线程安全，都支持阻塞式 `put/take`。  
2. 底层结构不同：`ArrayBlockingQueue` 基于数组，`LinkedBlockingQueue` 基于链表。  
3. 有界性不同：`ArrayBlockingQueue` 创建时必须指定容量（天然有界）；`LinkedBlockingQueue` 不指定容量时默认接近无界（`Integer.MAX_VALUE`），也可以手动指定容量变为有界。  
4. 锁模型不同：`ArrayBlockingQueue` 生产和消费共用一把锁；`LinkedBlockingQueue` 使用分离锁（`putLock` 和 `takeLock`），生产者与消费者并发时锁竞争通常更低。  
5. 内存特性不同：`ArrayBlockingQueue` 初始化时一次性分配数组空间；`LinkedBlockingQueue` 按节点动态分配，元素增长时内存逐步增加。  

### 关键词

1. 底层：数组 vs 链表  
2. 容量：必须有界 vs 可选有界（默认近无界）  
3. 锁：单锁 vs 双锁分离  
4. 内存：预分配 vs 动态分配  
5. 风险：`LinkedBlockingQueue` 不设容量可能堆积导致内存风险  

### 选型建议（面试可直接说）

1. 需要严格控制队列容量、明确背压：优先 `ArrayBlockingQueue`。  
2. 需要较高吞吐、生产消费并发明显：可考虑 `LinkedBlockingQueue`，但建议显式设置容量。
