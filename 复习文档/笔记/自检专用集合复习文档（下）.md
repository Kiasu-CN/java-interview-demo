# 自检专用集合复习文档（下）

## 1. HashMap 和 Hashtable 的区别 （因为HashTable现在很少使用，不是很重要）

### 答案（精简版）

1. 线程安全
- `HashMap` 线程不安全。
- `Hashtable` 线程安全（大量方法带 `synchronized`）。

2. 性能
- `HashMap` 通常更快。
- `Hashtable` 因同步开销较大，性能一般，实际开发基本不推荐。

3. `null` 支持
- `HashMap` 允许 `null` 键和 `null` 值（`null` 键只能一个）。
- `Hashtable` 不允许 `null` 键和 `null` 值，否则 `NullPointerException`。

4. 初始容量与扩容
- 默认初始容量：`HashMap` 常见是 `16`，`Hashtable` 是 `11`。
- 扩容规则：`HashMap` 常见 2 倍扩容；`Hashtable` 约 `2n+1`。
- 指定初始容量时：`HashMap` 会调整到 2 的幂；`Hashtable` 基本按给定值使用。

5. 底层结构与冲突优化
- JDK8+ `HashMap` 在链表过长时可树化（满足条件后转红黑树）。
- `Hashtable` 没有这套树化优化机制。

6. 哈希处理
- `HashMap` 对 `hashCode` 做了高低位扰动，降低冲突概率。
- `Hashtable` 主要直接使用 `hashCode`。

7. 实战建议
- 不建议新项目使用 `Hashtable`。
- 需要线程安全优先考虑 `ConcurrentHashMap`。

### 关键词

1. 线程安全：`HashMap` 不安全；`Hashtable` 安全（同步）
2. 性能：`HashMap` 更高
3. null：`HashMap` 允许；`Hashtable` 禁止
4. 容量：16 vs 11；2倍扩容 vs 2n+1
5. 结构：`HashMap`（JDK8+）链表可树化；`Hashtable` 无树化
6. 建议：并发场景优先 `ConcurrentHashMap`

### 对比速记

| 维度 | HashMap | Hashtable |
|---|---|---|
| 线程安全 | 否 | 是（synchronized） |
| 性能 | 较高 | 较低 |
| null 键值 | 允许 | 不允许 |
| 默认容量 | 16 | 11 |
| 扩容规则 | 约 2 倍 | 2n+1 |
| 冲突优化 | JDK8+ 可树化 | 无树化 |
| 现状 | 主流使用 | 基本淘汰 |

## 2. HashMap 和 TreeMap 的区别

### 答案（精简版）

1. 底层数据结构不同
- `HashMap` 基于 `数组 + 链表 + 红黑树`。
- `TreeMap` 基于 `红黑树`。

2. 是否有序不同
- `HashMap` 是无序的，不能保证遍历顺序。
- `TreeMap` 是有序的，默认按 `key` 的自然顺序排序，也可以自定义比较器排序。

3. 查询性能不同
- `HashMap` 平均时间复杂度通常是 `O(1)`。
- `TreeMap` 的增删改查通常是 `O(log n)`。

4. 对 `key` 的要求不同
- `HashMap` 主要依赖 `hashCode()` 和 `equals()`。
- `TreeMap` 依赖 `Comparable` 或 `Comparator` 来比较大小。

5. `null` 支持不同
- `HashMap` 允许一个 `null key`，也允许多个 `null value`。
- `TreeMap` 一般不允许 `null key`（自然排序下会抛异常），`null value` 通常可以。

6. 适用场景不同
- `HashMap` 适合普通的 KV 存储，重点是查询快。
- `TreeMap` 适合需要排序、范围查询、获取最大最小值等场景。

### 面试可直接回答（60秒）

`HashMap` 和 `TreeMap` 最大的区别是底层结构和是否有序。  
`HashMap` 底层是哈希表，JDK8 以后是数组、链表、红黑树结合，特点是无序，但查询效率高，平均是 `O(1)`。  
`TreeMap` 底层是红黑树，元素会按照 `key` 自动排序，所以增删改查一般是 `O(log n)`。  
另外，`HashMap` 主要依赖 `hashCode` 和 `equals`，而 `TreeMap` 依赖 `Comparable` 或 `Comparator`。  
如果只是做普通映射存储，优先用 `HashMap`；如果需要排序、区间查找、最值查找，就更适合 `TreeMap`。  

### 关键词

1. 结构：`HashMap` 哈希表；`TreeMap` 红黑树
2. 顺序：`HashMap` 无序；`TreeMap` 有序
3. 性能：`O(1)` vs `O(log n)`
4. 依据：`hashCode/equals` vs `Comparable/Comparator`
5. 场景：快速查询 vs 排序/范围查询

### 对比速记

| 维度 | HashMap | TreeMap |
|---|---|---|
| 底层结构 | 数组 + 链表 + 红黑树 | 红黑树 |
| 是否有序 | 无序 | 按 key 有序 |
| 查询效率 | 平均 `O(1)` | `O(log n)` |
| key 依据 | `hashCode` + `equals` | `Comparable` / `Comparator` |
| null key | 允许一个 | 通常不允许 |
| 典型场景 | 普通存储、快速查找 | 排序、范围查询、最值查询 |

## 3. HashMap 的底层实现

### 答案（精简版）

1. 整体来说，`HashMap` 底层本质是哈希表。
- 通过 `key` 的 `hashCode()` 计算哈希值，定位到数组下标。
- 如果多个元素落到同一个桶里，就发生哈希冲突。

2. `JDK1.8` 之前
- 底层结构是 `数组 + 链表`。
- 数组的每个位置叫一个桶（bucket）。
- 元素先通过哈希算法定位到桶，如果桶里为空就直接放入。
- 如果桶里已经有元素，就以链表形式挂在后面。
- 链表过长时，查找效率会下降到 `O(n)`。

3. `JDK1.8` 之后
- 底层结构是 `数组 + 链表 + 红黑树`。
- 大部分情况下还是先放到数组桶中。
- 如果某个桶中的链表长度过长，并且数组容量达到一定条件，会把链表转成红黑树。
- 这样是为了降低严重哈希冲突时的查找成本。

4. `JDK1.8` 为什么引入红黑树
- 因为链表太长时，查询效率低。
- 链表查询最坏是 `O(n)`，红黑树可以优化到 `O(log n)`。

5. 补充一点常见面试点
- `HashMap` 默认容量常见是 `16`，负载因子默认是 `0.75`。
- 当元素个数超过 `容量 * 负载因子` 时会触发扩容。
- 扩容通常是 `2` 倍。

### 面试可直接回答（60-90秒）

`HashMap` 底层本质上是哈希表，通过 `key` 的 `hashCode` 计算哈希，再定位到数组中的桶。  
如果不同的 key 算出来落到同一个桶，就会发生哈希冲突。  
在 `JDK1.8` 之前，`HashMap` 的结构主要是 `数组 + 链表`，冲突的元素会挂到同一个链表上，所以链表过长时查询效率会比较差，最坏是 `O(n)`。  
从 `JDK1.8` 开始，`HashMap` 优化成了 `数组 + 链表 + 红黑树`。当某个桶里的链表长度超过阈值，并且数组容量达到条件后，链表会树化成红黑树，这样最坏查询效率可以从 `O(n)` 优化到 `O(log n)`。  
所以可以理解为：`JDK1.8` 之前主要用链表解决冲突，`JDK1.8` 之后在冲突严重时会进一步转成红黑树来提高性能。  

### JDK1.8 前后对比

| 版本 | 底层结构 | 冲突处理 | 极端情况下查询效率 |
|---|---|---|---|
| `JDK1.8` 之前 | 数组 + 链表 | 链表挂接 | `O(n)` |
| `JDK1.8` 之后 | 数组 + 链表 + 红黑树 | 链表过长可树化 | `O(log n)` |

### 关键词

1. 本质：哈希表
2. 结构：数组
3. 冲突：链表法
4. `JDK1.8+`：引入红黑树
5. 目的：降低哈希冲突时的查找成本
6. 扩容：默认负载因子 `0.75`，通常 2 倍扩容

### 一句话速记

`HashMap` 底层是哈希表，`JDK1.8` 之前是 `数组 + 链表`，`JDK1.8` 之后是 `数组 + 链表 + 红黑树`，核心目的是在哈希冲突严重时提升查询效率。

## 4. HashMap 的 put 流程

### 答案（精简版）

1. 先对 `key` 做哈希计算
- `HashMap` 会先根据 `key` 的 `hashCode()` 计算哈希值。
- 再结合扰动函数，减少哈希冲突。

2. 计算桶下标
- 通过 `(n - 1) & hash` 计算元素应该落到数组哪个位置。

3. 如果数组还没初始化，先初始化
- 第一次 `put` 时，底层数组可能还没有创建。
- 这时会先进行初始化（懒加载）。

4. 如果对应桶为空，直接插入
- 如果这个位置没有元素，就直接放进去。

5. 如果桶里已经有元素，判断是否是同一个 key
- 先比较 `hash` 是否相同。
- 再比较 `key` 是否相等（`==` 或 `equals()`）。
- 如果是同一个 key，就覆盖旧值。

6. 如果不是同一个 key，就处理哈希冲突
- `JDK1.8` 之前通常采用头插法/链表处理冲突。
- `JDK1.8` 之后采用尾插法把节点挂到链表后面。
- 如果桶中已经是红黑树，就按红黑树方式插入。

7. 链表过长时，可能树化
- 在 `JDK1.8` 中，如果链表长度超过阈值，并且数组容量达到条件，链表会转成红黑树。

8. 插入完成后，判断是否需要扩容
- 元素个数增加后，如果超过阈值 `capacity * loadFactor`，就触发扩容。
- 扩容通常是 2 倍。

### 面试可直接回答（60-90秒）

`HashMap` 的 `put` 流程可以概括成：先算 hash，再找桶，冲突则覆盖或挂链，必要时树化，最后判断是否扩容。  
具体来说，先根据 `key` 的 `hashCode()` 计算哈希值，然后通过 `(n - 1) & hash` 定位到数组下标。  
如果数组还没初始化，会先初始化；如果对应桶为空，就直接插入。  
如果桶里已经有元素，会先判断是不是同一个 key，如果是就覆盖旧值；如果不是，就说明发生哈希冲突。  
这时在 `JDK1.8` 之后会把节点挂到链表尾部；如果桶已经树化了，就按红黑树方式插入。  
插入完成后，如果链表长度过长，可能会树化；如果元素总数超过阈值，还会触发扩容。  

### 流程速记

1. 算 `hash`
2. 算下标
3. 数组为空先初始化
4. 桶为空直接放
5. 桶不为空先判断是否同 key
6. 同 key 覆盖旧值
7. 不同 key 挂链或入树
8. 必要时树化
9. 超过阈值就扩容

### 高频追问补充

1. 为什么先比较 `hash` 再比较 `equals`
- 因为先比 `hash` 可以快速过滤，减少 `equals()` 调用成本。

2. 为什么 `JDK1.8` 改成尾插法
- 主要是为了在扩容迁移时避免链表逆序，也降低并发情况下形成环链表的问题。

3. 什么时候是替换，什么时候是新增
- 如果 `key` 相同，就是替换 value。
- 如果 `key` 不同但落到同一个桶，就是新增节点，属于哈希冲突。

### 一句话速记

`HashMap` 的 `put` 流程就是：`算 hash -> 定位桶 -> 空桶直接插入 -> 冲突则覆盖或挂链/入树 -> 最后判断树化和扩容`。

## 5. HashMap 冲突处理

### 答案（精简版）

1. 什么是哈希冲突
- 不同的 `key` 经过哈希计算后，落到了数组的同一个桶里，这就叫哈希冲突。

2. `HashMap` 怎么处理冲突
- 早期主要是 `链地址法`，也就是把冲突元素挂到同一个桶的链表上。
- `JDK1.8` 之后，如果链表过长，还会进一步转成红黑树。

3. `JDK1.8` 之前
- 底层是 `数组 + 链表`。
- 冲突后把新节点插入链表中。
- 链表越长，查找效率越低，最坏会退化到 `O(n)`。

4. `JDK1.8` 之后
- 底层是 `数组 + 链表 + 红黑树`。
- 冲突后先挂到链表上。
- 如果链表长度超过阈值，并且数组容量达到条件，就会树化成红黑树。
- 这样最坏查询效率可以优化到 `O(log n)`。

5. 为什么不一开始就用红黑树
- 因为大部分情况下冲突并不严重，链表结构更简单，维护成本更低。
- 只有链表过长时，才有必要转成红黑树优化性能。

### 面试可直接回答（60秒）

`HashMap` 处理哈希冲突的核心思路是链地址法。也就是说，不同的 key 如果算出来落到同一个桶里，不会直接覆盖，而是先挂到这个桶对应的数据结构上。  
在 `JDK1.8` 之前，主要是用链表来处理冲突，所以底层是 `数组 + 链表`；如果冲突很多，链表很长，查询效率最坏会退化到 `O(n)`。  
到了 `JDK1.8`，`HashMap` 做了优化，底层变成 `数组 + 链表 + 红黑树`。冲突后先挂链表，如果链表长度超过阈值，并且数组容量达到条件，就会把链表树化成红黑树，把最坏查询效率优化到 `O(log n)`。  
所以可以总结为：`HashMap` 先用链表解决冲突，冲突严重时再用红黑树进一步优化。  

### JDK1.8 前后区别

| 版本 | 冲突处理方式 | 极端查询效率 | 特点 |
|---|---|---|---|
| `JDK1.8` 之前 | 链表 | `O(n)` | 实现简单，但链表长时效率低 |
| `JDK1.8` 之后 | 链表 + 红黑树 | `O(log n)` | 冲突严重时性能更稳 |

### 关键词

1. 哈希冲突：不同 key 落到同一桶
2. 解决方式：链地址法
3. `JDK1.8` 前：链表
4. `JDK1.8` 后：链表过长树化
5. 目的：把最坏查询效率从 `O(n)` 优化到 `O(log n)`

### 一句话速记

`HashMap` 处理冲突的方式是链地址法，`JDK1.8` 之前主要靠链表，`JDK1.8` 之后链表过长会转成红黑树来优化查询效率。

## 6. 为什么 HashMap 的容量常用 2 的幂

### 先把符号说清楚

1. `n`：HashMap 当前数组容量（比如 16、32）。  
2. `hash`：key 的哈希值（一个 int）。  
3. `index`：元素落到数组里的下标。  
4. 下标计算：`index = (n - 1) & hash`。  

### 答案（易懂版）

1. 当 `n` 是 2 的幂时，`n - 1` 的二进制低位全是 `1`。  
2. 这时用 `&` 计算下标，结果一定落在 `0 ~ n-1`，刚好是数组合法下标范围。  
3. 同时这种位运算比取模更快，并且能较好利用 hash 的低位信息，让元素分布更均匀，减少冲突。  
4. 扩容时 HashMap 通常按 2 倍扩容，迁移规则简单：新位置要么是原位置，要么是 `原位置 + oldCap`，重排效率更高。  

### 为什么 `0 ~ n-1` 一定成立（最关键）

以 `n = 16` 为例：  
1. `n - 1 = 15`，二进制是 `1111`。  
2. `hash & 1111` 只会保留 `hash` 的低 4 位。  
3. 低 4 位的取值只能是 `0000 ~ 1111`，也就是十进制 `0 ~ 15`。  
4. 所以结果一定是合法下标。  

### 面试可直接回答（30秒）

HashMap 容量常用 2 的幂，是因为它用 `(n - 1) & hash` 算桶下标。  
当 `n` 是 2 的幂时，`n-1` 低位全 1，位运算结果天然落在 `0~n-1`，计算快、分布也更均匀。  
另外扩容按 2 倍时，元素迁移规则更简单（原位或原位+oldCap），重排成本更低。  

### 关键词

1. `n` 是容量，`hash` 是哈希值  
2. 下标公式：`(n - 1) & hash`  
3. 2 的幂 -> `n-1` 低位全 1  
4. 结果范围稳定在 `0~n-1`  
5. 扩容迁移：原位 / 原位 + oldCap

## 7. HashSet 去重

### 答案（精简版）

1. `HashSet` 底层其实是基于 `HashMap` 实现的。
- 往 `HashSet` 里加元素，本质上就是把元素当成 `HashMap` 的 `key` 来存。

2. 为什么能去重
- 因为 `HashMap` 的 `key` 不能重复。
- 如果两个对象的 `hashCode()` 相同，还会继续通过 `equals()` 判断是不是同一个对象。

3. 去重判断流程
- 先计算元素的 `hashCode()`。
- 定位到对应桶。
- 如果没有冲突，直接加入。
- 如果有冲突，再通过 `equals()` 比较。
- 如果 `equals()` 为 `true`，说明重复，不会再加入。
- 如果 `equals()` 为 `false`，说明不是同一个对象，可以共存在同一个桶里。

4. 面试中的关键点
- `HashSet` 去重不是只看 `hashCode()`。
- 真正判断重复要同时看 `hashCode()` 和 `equals()`。

### 面试可直接回答（60秒）

`HashSet` 去重的底层其实依赖的是 `HashMap`。因为 `HashSet` 内部会把存进去的元素当成 `HashMap` 的 key，而 `HashMap` 的 key 本身就不能重复，所以实现了去重。  
具体判断时，会先看元素的 `hashCode()`，定位到对应桶；如果桶里已经有元素，再通过 `equals()` 判断是否是同一个对象。  
如果 `hashCode` 相同并且 `equals()` 也返回 `true`，就认为是重复元素，不能加入；如果只是 `hashCode` 相同，但 `equals()` 不同，那只是哈希冲突，仍然可以共存。  
所以一句话说，`HashSet` 去重本质上依赖 `hashCode()` 和 `equals()` 共同决定。  

### 关键词

1. 底层：`HashMap`
2. 元素存储：元素作为 `key`
3. 去重依据：`hashCode()` + `equals()`
4. `hashCode` 相同不一定重复
5. `equals()` 为 `true` 才算真正重复

### 一句话速记

`HashSet` 底层用的是 `HashMap`，去重本质是利用 `HashMap key` 不可重复，判断依据是 `hashCode()` 和 `equals()`。

## 8. TreeMap / TreeSet 有序原理

### 答案（精简版）

1. `TreeMap` 和 `TreeSet` 为什么有序
- 因为它们底层都是基于 `红黑树` 实现的。
- 红黑树本身是一种自平衡二叉搜索树，节点会按大小关系组织。

2. 排序依据是什么
- 默认按元素的自然顺序排序，也就是实现 `Comparable`。
- 也可以在创建时传入 `Comparator`，按自定义规则排序。

3. `TreeMap` 和 `TreeSet` 的关系
- `TreeMap` 存的是键值对，按 `key` 排序。
- `TreeSet` 底层可以理解为基于 `TreeMap`，只存元素本身，元素本质上也是按 key 排序。

4. 为什么能一直保持有序
- 因为每次插入、删除元素时，红黑树都会调整结构，保持树的大致平衡和有序性。

### 面试可直接回答（60秒）

`TreeMap` 和 `TreeSet` 之所以有序，是因为它们底层都是基于红黑树实现的。红黑树本质上是一种自平衡的二叉搜索树，所以元素会按照大小关系组织起来。  
默认情况下，它们会按照元素的自然顺序排序，也就是要求 key 或元素实现 `Comparable`；如果没有自然顺序，也可以传入 `Comparator` 指定排序规则。  
其中 `TreeMap` 是对 `key` 排序，`TreeSet` 可以理解为只存 key 的有序集合。  
所以它们的“有序”本质不是插入顺序有序，而是按照比较规则排序。  

### 关键词

1. 底层：红黑树
2. 有序原因：二叉搜索树 + 自平衡
3. 默认排序：`Comparable`
4. 自定义排序：`Comparator`
5. `TreeMap` 按 key 排序，`TreeSet` 按元素排序

### 一句话速记

`TreeMap` 和 `TreeSet` 有序，是因为底层用红黑树实现，元素会按 `Comparable` 或 `Comparator` 的规则自动排序。

## 9. LinkedHashMap 特点

### 答案（精简版）

1. `LinkedHashMap` 是什么
- `LinkedHashMap` 可以看成 `HashMap + 双向链表`。
- 它既具备 `HashMap` 的哈希查找能力，又能维护元素顺序。

2. 它的主要特点
- 底层基于 `HashMap`，查询效率通常也比较高。
- 维护了一个双向链表，所以遍历顺序可控。
- 默认按插入顺序遍历。
- 也可以按访问顺序排列。

3. 为什么能保持顺序
- 因为每个节点除了哈希结构关系外，还额外维护了前驱和后继指针。
- 所以元素之间形成了一条双向链表。

4. 常见使用场景
- 需要保存插入顺序的 Map。
- 需要实现 LRU 缓存。

### 面试可直接回答（60秒）

`LinkedHashMap` 可以理解成 `HashMap` 的基础上又维护了一条双向链表。  
所以它既有 `HashMap` 查找快的特点，又可以保证遍历顺序。  
默认情况下，`LinkedHashMap` 是按插入顺序遍历的；如果开启了访问顺序模式，那么最近访问过的元素会被移动到链表尾部，这也是它常被用来实现 LRU 缓存的原因。  
所以它和 `HashMap` 的主要区别就是：`HashMap` 无序，而 `LinkedHashMap` 有顺序，并且这个顺序可以是插入顺序，也可以是访问顺序。  

### 对比速记

| 维度 | HashMap | LinkedHashMap |
|---|---|---|
| 底层结构 | 哈希表 | 哈希表 + 双向链表 |
| 遍历顺序 | 无序 | 有序 |
| 默认顺序 | 无 | 插入顺序 |
| 是否支持访问顺序 | 否 | 是 |
| 典型场景 | 普通 KV 存储 | 有序 Map、LRU |

### 关键词

1. 结构：`HashMap + 双向链表`
2. 特点：有序
3. 默认：插入顺序
4. 可选：访问顺序
5. 场景：LRU 缓存

### 一句话速记

`LinkedHashMap` 就是带双向链表的 `HashMap`，既能快速查找，又能维护插入顺序或访问顺序。

## 10. fail-fast / fail-safe

### 答案（精简版）

1. `fail-fast` 是什么
- 指在遍历集合时，如果检测到集合结构被修改，就立即失败并抛出异常。
- 常见异常是 `ConcurrentModificationException`。

2. `fail-safe` 是什么
- 指遍历时不是直接操作原集合，而是基于原集合的副本或弱一致性视图进行遍历。
- 所以遍历过程中即使集合被修改，也通常不会立刻抛 `ConcurrentModificationException`。

3. `fail-fast` 的典型代表
- `ArrayList`
- `HashMap`
- `HashSet`
- 大多数普通集合类的迭代器

4. `fail-safe` 的典型代表
- `CopyOnWriteArrayList`
- `ConcurrentHashMap`
- 这类并发容器更适合并发场景下遍历

5. 两者核心区别
- `fail-fast` 是“发现有人改了，立刻报错”。
- `fail-safe` 是“遍历自己的副本/弱一致视图，不和原集合强绑定”。

6. 注意点
- `fail-fast` 不是线程安全机制，只是尽最大努力快速报错。
- `fail-safe` 虽然遍历更安全，但可能读到的不是最新数据。

### 面试可直接回答（60秒）

`fail-fast` 和 `fail-safe` 说的是集合在遍历过程中，面对并发修改时的不同处理策略。  
`fail-fast` 的意思是快速失败，也就是迭代时如果发现集合结构被修改了，就立刻抛出 `ConcurrentModificationException`。像 `ArrayList`、`HashMap` 这类普通集合的迭代器，大多都是这种机制。  
`fail-safe` 的意思是安全失败或者说安全遍历，它通常不是直接遍历原集合，而是遍历副本，或者使用弱一致性视图，所以在遍历时即使集合被修改，也一般不会立刻抛异常。像 `CopyOnWriteArrayList`、`ConcurrentHashMap` 更接近这种机制。  
所以两者的本质区别是：`fail-fast` 发现修改就报错，`fail-safe` 允许遍历和修改并存，但可能读到旧数据或非实时数据。  

### 对比速记

| 维度 | fail-fast | fail-safe |
|---|---|---|
| 遍历对象 | 原集合 | 副本或弱一致视图 |
| 修改时表现 | 立即报错 | 通常不报错 |
| 常见异常 | `ConcurrentModificationException` | 一般无此异常 |
| 代表集合 | `ArrayList`、`HashMap` | `CopyOnWriteArrayList`、`ConcurrentHashMap` |
| 数据一致性 | 更强调及时发现错误 | 更强调遍历可继续进行 |

### 关键词

1. `fail-fast`：快速失败
2. `fail-safe`：安全遍历
3. 异常：`ConcurrentModificationException`
4. 前者遍历原集合，后者遍历副本/弱一致视图
5. 前者容易报错，后者可能读到旧数据

### 一句话速记

`fail-fast` 是遍历时发现集合被改了就立刻报错，`fail-safe` 是遍历副本或弱一致视图，所以一般不报错但可能读不到最新数据。
