# 算法题自检清单

目标：减少“思路对了但代码错了”的情况，尤其适用于二分、双指针、贪心、动态规划、链表等高频题型。

使用方式：每做完一道题，不管 AC 没 AC，都按这份清单快速过一遍。前期可以逐条检查，后期压缩成 20-30 秒的心智流程。

---

## 一、先问自己：题目允许空结果吗

这是很多初始化错误的根源。

常见判断：

- 子数组是否要求至少包含一个元素
- 子序列是否可以为空
- 树节点、链表节点是否可能为 `null`
- 查找失败时是否明确要求返回 `-1`
- 是否允许“不选任何元素”

典型错误：

- `LC53 最大子数组和` 里把 `max` 初始化成 `0`

为什么会错：

- 这相当于默认“空子数组”的和也可以参与比较
- 但题目要求子数组至少包含一个元素

结论：

- 只要题目要求“必须选一个真实元素”，初始化通常就不能随便写 `0`

---

## 二、每个变量到底表示什么

写代码前，必须能用一句完整的话解释每个核心变量。

比如：

- `left` / `right`：当前搜索区间的左右边界
- `sum`：以当前位置结尾的最大子数组和
- `max`：遍历到当前位置为止的全局最大值
- `dp[i]`：到 `i` 为止的最优解，或者以 `i` 结尾的最优解

如果变量定义不清楚，就很容易出现这些问题：

- 初始化乱写
- 更新顺序写反
- 循环结束后不知道答案该返回谁

自检问题：

- 这个变量表示的是“真实结果”，还是“辅助状态”？
- 它表示“全局最优”，还是“局部最优”？
- 它表示“到当前位置为止”，还是“必须以当前位置结尾”？

---

## 三、初始化是否和变量定义一致

很多题不是思路错，而是初始化把后面的正确逻辑带歪了。

判断方法：

- 如果变量代表“某个真实解”，初值通常应该来自真实输入
- 如果变量只是“计数器/辅助量”，才更可能从 `0` 开始

例如：

```java
int sum = nums[0], max = nums[0];
```

这比下面这版更符合 `LC53` 的题意：

```java
int sum = 0, max = 0;
```

因为：

- `sum` 和 `max` 表示的都是“真实子数组的和”
- 既然是真实子数组，就应该从真实元素开始

---

## 四、边界条件和区间定义是否成套

这一点在二分查找里最容易出错。

### 1. 左闭右闭区间 `[left, right]`

对应写法：

```java
int left = 0, right = nums.length - 1;
while (left <= right) {
    int mid = left + (right - left) / 2;
    if (nums[mid] == target) {
        return mid;
    } else if (nums[mid] < target) {
        left = mid + 1;
    } else {
        right = mid - 1;
    }
}
```

关键点：

- `right = nums.length - 1`
- `while (left <= right)`
- `left == right` 时最后一个元素仍然要检查

### 2. 左闭右开区间 `[left, right)`

对应写法：

```java
int left = 0, right = nums.length;
while (left < right) {
    int mid = left + (right - left) / 2;
    if (nums[mid] < target) {
        left = mid + 1;
    } else {
        right = mid;
    }
}
```

关键点：

- `right = nums.length`
- `while (left < right)`
- `right` 本身不参与搜索

结论：

- 不要单独背某一行代码
- 要整套记住“区间定义 + 循环条件 + 边界更新”

---

## 五、主动拿极端样例卡自己

很多错误不是推理不出来，而是没有主动测最容易翻车的输入。

每道题至少过这几类样例：

- 只有一个元素
- 全是负数
- 全是正数
- 含有 `0`
- 答案在开头
- 答案在结尾
- 找不到目标
- 输入为空或为 `null`（如果题目允许）

例如：

### `LC53 最大子数组和`

必须至少手测：

```java
[-1]
[-2, -1, -3]
[1]
[5, 4, -1, 7, 8]
```

### `LC704 二分查找`

必须至少手测：

```java
[5], target = 5
[5], target = 1
[1, 3, 5], target = 5
[1, 3, 5], target = 2
```

如果这些样例没过，往往说明：

- 初始化有问题
- 循环条件有问题
- 最后一个位置漏检查了

---

## 六、循环结束时，最后一种情况处理了吗

很多题不是中间逻辑错，而是结束时漏了一步。

常见漏法：

- 二分里 `left == right` 没检查
- 滑动窗口最后一段答案没结算
- 链表最后一个节点没处理
- 遍历过程中答案更新顺序写反

自检问题：

- 循环退出时，是否还有一个合法状态没被处理
- 我是先更新答案，再移动指针，还是顺序写反了
- 最后一个元素、最后一个区间、最后一个节点是否被覆盖到

---

## 七、做完题后的 20 秒复盘模板

可以直接按下面这 4 句过脑子：

1. 这题允不允许空结果
2. 每个核心变量到底表示什么
3. 初始化有没有和变量定义、题目要求对齐
4. 我有没有拿极端样例卡边界

如果想再稳一点，就补一句：

5. 循环退出时，最后一个合法情况有没有漏掉

---

## 八、结合常见错误看自己的薄弱点

### 1. 二分写成 `left < right`

暴露的问题：

- 区间定义和循环条件没有配套

本质原因：

- 没有先想清楚当前维护的是 `[left, right]` 还是 `[left, right)`

### 2. 最大子数组和写成 `max = 0`

暴露的问题：

- 初始化没有和“至少选择一个元素”对齐

本质原因：

- 默认把 `0` 当成安全答案
- 没有主动验证“全负数”这种极端情况

---

## 九、以后刷题时最该养成的习惯

- 先定义变量含义，再写代码
- 先确定区间语义，再写循环条件
- 先想极端样例，再相信自己的代码
- 先检查初始化是否符合题意，再检查转移逻辑

一句话记忆：

**定义 -> 初始化 -> 边界 -> 极端样例**

这 4 步如果形成习惯，很多“差一点就对”的错误都会明显减少。
