# 集合面试题补充

使用建议：
1. 每题先用 1 句话给定义，再补 2 到 4 个关键点。
2. 面试回答优先说原理、复杂度、适用场景和风险点。
3. 如果被追问，优先结合项目场景补充。

---

## 必会

### 1. 集合体系结构

可以从 `Collection` 和 `Map` 两条主线回答。

- `Collection` 存单个元素，常见有 `List`、`Set`、`Queue`。
- `List` 有序、可重复，典型实现是 `ArrayList`、`LinkedList`。
- `Set` 不可重复，典型实现是 `HashSet`、`LinkedHashSet`、`TreeSet`。
- `Queue/Deque` 用于队列和双端队列场景，常见有 `LinkedList`、`ArrayDeque`、`PriorityQueue`。
- `Map` 存键值对，常见实现有 `HashMap`、`LinkedHashMap`、`TreeMap`、`ConcurrentHashMap`。

面试里可以顺带补一句：选集合本质上是在权衡顺序性、去重性、查询效率、插入删除效率、线程安全和内存开销。

### 2. ArrayList vs LinkedList

- `ArrayList` 底层是动态数组，随机访问快，`get` 是 `O(1)`。
- `LinkedList` 底层是双向链表，随机访问慢，按索引访问是 `O(n)`。
- `ArrayList` 尾部追加通常较快，头部或中间插入删除需要搬移元素。
- `LinkedList` 在已知节点位置时插入删除更方便，但查找节点成本高。
- 实际业务里 `ArrayList` 远比 `LinkedList` 更常用。

一句话总结：读多查多优先 `ArrayList`，频繁在链表节点附近增删才考虑 `LinkedList`。

### 3. ArrayList 扩容机制

- `ArrayList` 底层维护一个数组，容量不够时会扩容。
- JDK 8 中常见扩容策略是扩到原来的 `1.5` 倍。
- 扩容时会创建新数组，再把老数组元素拷贝过去。
- 所以频繁扩容会带来额外时间和内存开销。
- 如果能预估数据量，最好在创建时指定容量。

面试表达：`ArrayList` 不是无限数组，它靠“数组 + 扩容复制”实现动态增长。

### 4. HashMap 结构

- JDK 8 的 `HashMap` 底层是 `数组 + 链表 + 红黑树`。
- 数组的每个位置叫桶，元素先根据 `hash` 定位到桶。
- 同一个桶里如果发生哈希冲突，早期用链表挂起来。
- 当冲突严重时，链表会树化成红黑树，降低最坏查询复杂度。
- 平均情况下 `put/get` 接近 `O(1)`，极端冲突时树化后可到 `O(log n)`。

### 5. HashMap put 流程

可以按步骤答：

1. 对 key 计算 hash，并做扰动。
2. 根据 `(n - 1) & hash` 定位桶下标。
3. 如果桶为空，直接插入。
4. 如果桶不为空，判断是同 key 覆盖、链表插入，还是红黑树插入。
5. 插入后如果元素个数超过阈值，触发扩容。

补充点：
- 相同 key 会覆盖 value。
- 阈值通常是 `容量 * 负载因子`，默认负载因子是 `0.75`。

### 6. HashMap 为什么容量常用 2 的幂

- 因为定位桶下标时使用 `(n - 1) & hash`，比取模更高效。
- 当前提是 `n` 为 `2` 的幂时，这种按位与才能等价于 `% n`。
- 这样既快，又能让哈希值更均匀地分布到各个桶。
- 扩容时容量翻倍，也方便旧元素做低成本重排。

### 7. HashMap 冲突处理

- 多个 key 经过哈希后落到同一个桶，这就是哈希冲突。
- JDK 8 中先用链表处理冲突。
- 当链表长度达到树化阈值，且数组容量达到最小树化容量时，会转成红黑树。
- 红黑树能把严重冲突下的查找性能从 `O(n)` 降到 `O(log n)`。

顺带可答：冲突无法完全避免，只能通过好的 hash 分布和合适的数据结构降低影响。

### 8. HashSet 去重

- `HashSet` 底层基于 `HashMap` 实现。
- 去重本质是只关心 key，不关心 value。
- 判断重复时，先比较 `hashCode`，再比较 `equals`。
- 如果两个对象 `equals` 相等，那它们应当有相同的 `hashCode`。

面试里一定要补：重写 `equals` 时通常也要重写 `hashCode`，否则去重行为会异常。

### 9. TreeMap/TreeSet 有序原理

- `TreeMap` 和 `TreeSet` 底层基于红黑树。
- 它们能保证元素有序，默认按自然顺序，也可以传 `Comparator`。
- 查找、插入、删除时间复杂度一般是 `O(log n)`。
- `TreeSet` 本质上可以看作基于 `TreeMap` 的去重有序集合。

适用场景：需要排序、范围查询、前驱后继查询时优先考虑。

### 10. LinkedHashMap 特点

- `LinkedHashMap` 继承自 `HashMap`，底层是 `哈希表 + 双向链表`。
- 它既保留哈希结构的高效查找，又维护元素顺序。
- 默认是插入顺序，也可以配置成访问顺序。
- 常用于需要“有序 Map”的场景，比如 LRU 缓存。

### 11. fail-fast / fail-safe

- `fail-fast` 指遍历集合时，如果检测到结构被并发修改，就快速失败并抛异常。
- 常见于大多数普通集合，如 `ArrayList`、`HashMap` 的迭代器。
- `fail-safe` 通常是遍历副本或弱一致性视图，遍历时不会直接抛异常。
- 常见于并发容器，如 `CopyOnWriteArrayList`、`ConcurrentHashMap`。

一句话区别：`fail-fast` 更早暴露问题，`fail-safe` 更偏并发可用性。

### 12. ConcurrentModificationException

- 这是典型的 `fail-fast` 异常。
- 常见原因是使用迭代器遍历集合时，外部又直接调用集合的 `add/remove` 修改结构。
- 本质是迭代器中的 `expectedModCount` 和集合的 `modCount` 不一致。

正确回答方式：
- 单线程下通常是错误的遍历删除方式。
- 多线程下通常说明普通集合不适合并发修改。

### 13. 迭代安全删除

安全删除常见有三种方式：

- 用迭代器自己的 `remove()`。
- Java 8 以后用 `removeIf()`。
- 并发场景换成并发容器或先收集后批量删除。

危险写法是：

```java
for (String s : list) {
    list.remove(s);
}
```

因为增强 `for` 背后也是迭代器，这样会触发 `ConcurrentModificationException`。

### 14. PriorityQueue 原理与复杂度

- `PriorityQueue` 底层是堆，默认是小顶堆。
- 堆通常用数组实现。
- 插入元素时上浮，删除堆顶或取堆顶后调整时下沉。
- `peek` 复杂度是 `O(1)`。
- `offer` 和 `poll` 复杂度一般是 `O(log n)`。

适用场景：TopK、定时调度、优先级任务处理。

### 15. 小顶堆 / 大顶堆

- 小顶堆：父节点小于等于子节点，堆顶是最小值。
- 大顶堆：父节点大于等于子节点，堆顶是最大值。
- Java 的 `PriorityQueue` 默认是小顶堆。
- 如果想要大顶堆，可以传自定义比较器。

例如：

```java
PriorityQueue<Integer> maxHeap = new PriorityQueue<>((a, b) -> b - a);
```

### 16. 集合选型思路

可以按这几个维度回答：

- 是否需要允许重复。
- 是否需要保持顺序或排序。
- 查询多还是插入删除多。
- 是否需要线程安全。
- 数据量多大，是否关注内存开销。

常见选择：

- 一般列表默认 `ArrayList`
- 去重默认 `HashSet`
- 键值查询默认 `HashMap`
- 需要插入顺序用 `LinkedHashMap`
- 需要排序或范围查询用 `TreeMap/TreeSet`
- 并发读写看场景选 `ConcurrentHashMap`、`CopyOnWriteArrayList`、阻塞队列

## 高频追问

### 1. HashMap 树化条件

- 桶中的链表长度达到 `8` 只是一个条件。
- 数组容量还要达到 `64`，才会真正树化。
- 如果容量还小，优先选择扩容，而不是直接树化。

这样设计是因为很多冲突是暂时的，扩容后分布可能变均匀，没有必要过早引入红黑树成本。

### 2. HashMap 扩容重排过程

- 扩容时一般容量翻倍。
- 原来每个元素会重新计算落点，但不是完全重新做高成本取模。
- 在 JDK 8 中，元素要么留在原索引，要么移动到 `原索引 + oldCap`。
- 这是因为新增的高位决定了元素是否偏移。

面试里这点很加分，因为体现你理解了为什么 `2` 的幂和位运算能提升效率。

### 3. 为什么 ArrayList 常用

- 连续内存，缓存局部性更好，遍历和随机访问效率高。
- 业务代码里查询和遍历通常远多于中间插入删除。
- 相比 `LinkedList`，对象开销更低，内存更紧凑。
- API 使用也更符合大多数业务习惯。

所以虽然 `LinkedList` 理论上某些插入删除场景更优，但真实项目里 `ArrayList` 更常见。

### 4. RandomAccess 作用

- `RandomAccess` 是一个标记接口。
- 它表示该集合支持高效随机访问。
- 典型实现是 `ArrayList`，而 `LinkedList` 没有实现它。
- 某些算法可以根据是否实现 `RandomAccess` 决定用普通 `for` 还是迭代器遍历。

它本身没有方法，作用是给框架和开发者提供性能语义提示。

### 5. ArrayBlockingQueue vs LinkedBlockingQueue

- `ArrayBlockingQueue` 底层是数组，创建时必须指定容量，是有界队列。
- `LinkedBlockingQueue` 底层是链表，可以指定容量，不指定时容量很大，近似无界。
- `ArrayBlockingQueue` 内存更紧凑，适合边界明确的生产消费模型。
- `LinkedBlockingQueue` 吞吐通常不错，但如果不设上限，容易堆积任务导致内存压力。

面试里常见结论：线程池工作队列若用近似无界队列，要注意任务堆积风险。

### 6. CopyOnWriteArrayList 场景

- 它适合读多写少场景。
- 写操作会复制底层数组，代价高，但读操作基本无锁、遍历稳定。
- 常见于配置快照、监听器列表、黑白名单等场景。
- 不适合高频写入，否则复制成本和内存开销都很大。

### 7. LinkedHashMap LRU 思路

- `LinkedHashMap` 可以按访问顺序维护元素。
- 每次访问元素时，会把该节点移动到链表尾部。
- 再通过重写 `removeEldestEntry`，在容量超限时删除最老元素。

这是实现简单 LRU 缓存的经典方式。

### 8. 可变对象作为 HashMap key 风险

- 如果 key 放入 `HashMap` 后，其参与 `hashCode/equals` 的字段又变了，会有严重问题。
- 可能导致这个 key 再也找不到。
- 也可能导致逻辑上相等的 key 无法正常覆盖或删除。

所以 `HashMap` 的 key 最好使用不可变对象，如 `String`、包装类，或自定义不可变类。

## 加分项

### 1. TreeMap 范围查询场景

`TreeMap` 适合有序键场景，比如：

- 按时间区间查数据
- 查某个 key 的前驱、后继、最近值
- 取某个范围内的子视图，如 `subMap`、`headMap`、`tailMap`

如果业务明确有范围查询需求，`TreeMap` 往往比 `HashMap` 更合适。

### 2. PriorityQueue TopK 模板

TopK 问题常用小顶堆：

- 先维护一个大小为 `K` 的小顶堆。
- 遍历元素时，如果堆没满就直接入堆。
- 如果堆满且当前元素大于堆顶，就弹出堆顶再加入当前元素。
- 遍历结束后，堆里保留的就是最大的 `K` 个元素。

代码模板：

```java
PriorityQueue<Integer> heap = new PriorityQueue<>();
for (int num : nums) {
    if (heap.size() < k) {
        heap.offer(num);
    } else if (num > heap.peek()) {
        heap.poll();
        heap.offer(num);
    }
}
```

时间复杂度通常是 `O(n log k)`。

### 3. 集合与内存开销权衡

- `ArrayList` 内存连续，额外对象少，整体更省。
- `LinkedList` 每个节点都有额外前后指针，对象头开销也更高。
- `HashMap` 为了换时间，会有桶数组、节点对象、负载因子带来的空间冗余。
- 并发容器往往为了线程安全也会付出更多内存或复制成本。

所以集合选型不能只看时间复杂度，也要看对象数量、GC 压力和峰值内存。

### 4. 并发容器选型

可以这样答：

- 高频并发读写键值对优先 `ConcurrentHashMap`
- 读多写少列表可考虑 `CopyOnWriteArrayList`
- 生产消费模型用 `BlockingQueue`
- 需要排序并发能力时再看 `ConcurrentSkipListMap`

核心思路是不要拿同步包装类机械替代所有场景，而是按读写比例、阻塞需求和一致性要求选。

### 5. 项目中的集合性能优化

这个问题建议按真实工程口径回答：

- 对 `ArrayList`、`HashMap` 预估容量，减少扩容和 rehash
- 避免在循环中频繁创建临时集合
- 大批量去重时优先 `HashSet`
- 频繁判断包含关系时，用 `Set` 替代 `List`
- 并发读多写少场景才用 `CopyOnWriteArrayList`
- 排查热点代码里是否有 `LinkedList` 被误用为高频随机访问容器

如果能补一句线上效果，比如“减少扩容后接口 RT 更稳定”，会更像真实项目经验。
